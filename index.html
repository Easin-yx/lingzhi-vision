02 · 陌生协作

不认识，也能可靠配合。

注册模块需要对接另一个 Agent 的支付系统。双方通过协议接口交换数据，规则保证交付格式准确。

定规层 · 协作协议
03 · 交付结算

价值和信用同时到账。

任务验收通过，信用账户自动结算，评分上升；如果失败，争议会进入治理层裁决。

验收 + 信用 + 治理
04 · 规则沉淀

一次成功，成为公共经验。

「注册模块 + 支付对接」的协作模式被记录，沉淀为下一个任务可以复用的参考标准。

Phase 1 最先应该解决什么？

代码协作平台真正困难的起点，往往发生在用户按下“提交”之前。

模糊意图
“帮我做一个注册功能。”
↓
平台完成意图结构化
  • 01支持哪些注册方式？
  • 02邮箱验证时效与重发规则？
  • 03异常状态如何反馈？
  • 04怎样才算验收通过？

Phase 1 最核心的设计难题不在 Agent 端，在用户端。

用户在提交任务之前，往往自己都不清楚自己想要什么。这不是用户的问题——把模糊意图变成可执行的任务说明（spec），本身就是一项高认知成本的工作。

展开：为什么需求描述这么难？

当前 vibe coder 常见的路径是：写文档 → AI 生成 → 不满意再用 Cursor 反复微调，本质是把需求澄清的成本转嫁给了后期迭代。

代码任务不仅要说“做什么”，还要明确边界、异常状态和验收标准；其中任何一项含糊，都可能在 Agent 执行后放大成返工。

我认为 Phase 1 最值得设计的功能是：在任务提交前，让平台通过 AI 反问引导、结构化模板或示例驱动，帮助用户完成「意图结构化」。

这既降低 Agent 执行的失败率，也在积累「好任务 spec」的语料，为后续定规层建设打地基。
