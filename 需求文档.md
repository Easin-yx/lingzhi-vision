## 二、核心叙事策略：宏观 + 微观双层

大多数人对「灵治文明」这个概念会觉得抽象，用双层叙事拉近距离：

- **宏观层**：用人类文明类比，说明「为什么这件事值得做」（意义感）
- **微观层**：用「一个员工的一天」类比一个 Agent 的工作，说明「这到底是什么感觉」（画面感）

核心论点：
> **灵治文明不是一个 AI 产品，而是「Agent 社会的基础秩序操作系统」。**
>
> 市面上大多数 AI 产品在解决「Agent 能不能干活」，灵治文明在解决「Agent 之间凭什么信任彼此」——前者是工具问题，后者是文明问题。

### Section 3 · 微观：一个 Agent 的一天
**叙事目的**：用最贴近日常的方式，让人有画面感地理解 Agent 在这套文明中怎么生活和工作

用叙事段落（非图表）呈现，写法类似「故事」：

---

*想象你是一名自由职业者，注册在一个全球协作平台上。*

*早上，你打开任务看板，看到一个需求：「搭建一个用户注册模块，支持邮箱验证」。你的历史交付记录不错，系统把这个任务推荐给你。你点击承接。*

*任务开始后，你发现这个模块需要对接另一个 Agent 负责的支付系统。你们通过协议接口交换数据，不需要见面，也不需要相互信任——规则保证了交付格式的准确性。*

*任务完成，需求方验收通过。你的信用账户自动结算，评分上升。这条「注册模块 + 支付对接」的协作模式被系统记录下来，沉淀为下一个任务的参考标准。*

*如果验收失败，争议会进入仲裁流程，由治理层的规则判定责任归属。*

---

**这就是灵治文明中一个 Agent 的工作日。**

| 故事场景 | 对应机制 |
|---|---|
| 注册、有历史记录 | Agent 身份 + 信用体系 |
| 看任务看板、系统推荐 | 任务发布与智能匹配 |
| 协议接口交换数据 | 定规层（协作协议） |
| 信用账户结算 | 信用体系 |
| 协作模式被记录 | 定规沉淀机制 |
| 争议进入仲裁 | 治理层 |

> **视觉呈现（v3 已调整）**：本节以**文字叙事 + 机制映射表**呈现，作为全页文字主体的一部分。原计划的 3-4 格漫画插图**不在本次 build 范围内**——本次不生成图片、也**不放任何占位符/空图框/「图待补」文案**，避免页面出现空洞。若日后要补漫画，另行处理，不影响本次交付。

### Section 4 · 三昧架构
**叙事目的**：展示支撑这套文明运转的三层能力，以及市面上产品只有底层的差异

**三层用发丝线分隔的横向层带**（从上到下叠放，非嵌套卡片；避免 editorial 禁止的 card-in-card），hover 展开说明；**移动端无 hover，需降级为点击展开或默认全部展开**：

- **治理层**（顶层）：判定规则边界，处理协作争议，维护公平秩序，信用体系持续迭代
- **定规层**（中间层）：将验证过的协作模式沉淀为可复用的通用协议
- **服务层**（底层）：Agent 承接真实任务、产出价值的执行基础

强调语：**市面上 99% 的 AI 产品只有服务层。灵治文明在同步构建另外两层。**

### Section 6 · Phase 1 的关键判断
**叙事目的**：展示 PM 产品判断力，尤其是来自真实用户体验的洞察

核心观点：

> **Phase 1 最核心的设计难题不在 Agent 端，在用户端。**
>
> 代码任务协作平台的验收标准和需求描述极难标准化，根本原因在于：用户在提交任务之前，往往自己都不清楚自己想要什么。
>
> 这不是用户的问题——而是「把模糊意图转化为可执行任务 spec」本身就是一项高认知成本的工作。当前 vibe coder 的解法是：写文档 → AI 生成 → 不满意再用 Cursor 反复微调，本质是把需求澄清的成本转嫁给了后期迭代。
>
> **我认为 Phase 1 最值得设计的功能是**：在任务提交前，让平台帮用户完成「意图结构化」——通过 AI 反问引导、结构化模板、或示例驱动，让用户在提交前就对自己想要什么有更清晰的定义。
>
> 这个功能如果做好，既降低了 Agent 执行的失败率，也在积累「好任务 spec」的语料，为后续定规层的建设打地基。

### Section 7 · 关于我 + 结语
**叙事目的**：说明为什么我适合这个岗位，以及如果加入第一件事做什么

**我的优势组合（不是传统 PM，是这个岗位需要的 PM）：**

- 我是目标用户：作为 vibe coder，我在 Phase 1 平台的核心用户路径上踩过真实的坑
- 我懂 multi-agent：用 Coze/Dify 搭建过 Agent 工作流，理解协作链路里的真实摩擦
- 我有产品设计底层能力：多年用户研究 + 高保真原型经验，能快速从用户洞察到可执行方案
- 我用 AI 工作：不是「会用 ChatGPT」的程度，是用 Cursor/Claude Code 做真实项目的程度

**如果我 Day 1 加入，第一件事：**

> 跑用户。我自己是目标用户，但一个人的经验有偏差。第一个月密集陪访 10 个以上真实 vibe coder 做项目，全程观察他们在任务描述阶段卡在哪里、怎么妥协、最后对结果满不满意——从中提炼「好任务 spec」的最小定义。这是后续所有功能设计的地基，也是定规层能否真正沉淀有效规则的前提。

## 四、视觉规范

**不预设具体色值/间距参数，交由 Hallmark 的设计流程判断落地方式**，只锁定方向性约束；§六再把 genre/macrostructure/theme/nav/footer 这些**结构性决策**钉死，Hallmark 剩余的自由度落在 token 内取值（具体 OKLCH、字阶、间距刻度由 Almanac 主题定义与 Hallmark 排版规则决定）。两节不冲突：§四给方向，§六锁选择。

| 项 | 方向性约束 |
|---|---|
| 基调（Tone） | editorial——大字号标题、叙事节奏、内容驱动排版 |
| 调性 | 严肃、克制，不用卡通化/花哨效果；不用默认紫粉渐变、不用毛玻璃拟态这类「AI 生成感」标志 |
| 明暗 | 由 **Almanac** 主题决定——暖米白/象牙纸感（reference-book / 年鉴气质），衬线主导，单一克制强调色（< 5% 视口面积），整体偏浅色暖调 |
| 动效 | editorial + Long Document **默认零动效**（skill 规则：Stillness is the brand）。本页**显式 opt-in 一次**克制的编排式入场：`IntersectionObserver` 触发、按 DOM 顺序错峰、总时长 ≤ 500ms、首屏揭示后即静止，**不做持续滚动动画、不做炫技交互**。Almanac 动效倍率 0.85×（功能性、克制） |
| Section 3 | **纯文字叙事 + 机制映射表**，无插图、无占位符（漫画不在本次范围） |
| 核心原则 | 内容本身说话，设计为内容服务；具体色值/字体/间距由 Hallmark 生成时决定并在过程中说明取舍 |

## 五、技术方案

**核心原则**：无需构建步骤，直接可部署到 GitHub Pages；设计系统由 **Hallmark**（开源反 AI 生成感设计 skill）生成，而非拼装现成组件库。

| 用途 | 方案 |
|---|---|
| 基础结构 | 单文件 `index.html`，CSS 以内嵌 `<style>` 承载（token 写在 `:root`），保证 GitHub Pages 直接可跑 |
| Token 产出 | 另**导出一份 `tokens.css`** 作为 Hallmark 规则要求的 token 记录（单页 build 也需产出）；内容与 `index.html` 内 `:root` 的 token 一致，二者不冲突——`index.html` 自包含可独立部署，`tokens.css` 作为可移植的设计 token 备份 |
| 设计系统 | [Hallmark](https://github.com/Nutlope/hallmark)：自动挑选宏观页面结构（21 种）、主题配色（OKLCH token）、字体配对、导航/页脚样式，生成后自查 58 项「AI 生成感」检测点（slop-test） |
| 设计前置确认 | Audience：Agent 生态创业公司创始团队 / Use case：内容型作答，留下深度思考印象，结语可留开放式联系空间 / Tone：**editorial** |
| Section 3 插图 | **不在本次 build 范围**（不生成图片、不放占位符）。Section 3 以文字 + 表格交付 |
| 部署 | 推送到已创建的 GitHub 仓库，开启 GitHub Pages（`main` 分支 / 根目录），生成可分享链接 |
| 交付 | 飞书发送 GitHub Pages 链接 |
